# Backend Implementation Order: Live Event Agentic Assistant

## Goal
Build the data pipeline, transcript ingestion, and LLM analysis logic for the real-time speaker HUD. The backend processes event data and returns structured commands to the Streamlit UI.

For UI-specific steps, see `frontendOrder.md`. For shared contracts and state management, see `integrationOrder.md`.

## Backend / LLM Engineer Responsibilities

- Load and normalize attendee data from `hackathon_mock_attendees.xlsx`.
- Load and normalize session metadata.
- Connect the live transcription source to the analysis pipeline.
- Build the audience profile for the active session.
- Build prompt inputs from rolling transcript context and audience context.
- Run time-slotted LLM analysis over the live transcript flow.
- Implement LLM response parsing into simple UI-ready commands.
- Provide stable functions the UI can call.

## Implementation Order

### 1. Data Discovery and File Loading

Tasks:
- Locate the attendee Excel file and session metadata.
- Implement data loading functions using `pandas` for Excel and standard JSON parsing for metadata.
- Normalize column names and metadata keys into predictable internal names.
- Add defensive handling for missing or malformed files.

Suggested functions:

```python
load_attendees(path) -> pandas.DataFrame
load_sessions(path_or_dir) -> list[dict]
normalize_transcript_chunk(raw_chunk) -> dict
```

### 2. Session Selection and Audience Profiling

Tasks:
- Filter registered attendees for the selected room/session when the data contains session or room registration fields.
- Calculate audience profile fields:
  - attendee count
  - AI experience distribution
  - academic background distribution
  - top attending intents
  - beginner ratio
- Return the profile using the shared `Audience Profile` shape defined in `integrationOrder.md`.

### 3. Live Transcript Ingestion

Tasks:
- Connect to the real-time transcript source used during the hackathon.
- Normalize each incoming transcript update into the shared `Transcript Chunk` shape.
- Maintain a rolling transcript window in memory.
- Store latest transcript state in a lightweight in-process queue or Streamlit session state.
- Add clear handling for disconnected, waiting, and receiving states.

Done when:
- The UI can receive and display transcript chunks.
- The latest transcript chunk is visible.
- The rolling transcript window is available to the analysis layer.

### 4. Neutral Fallback

The live analysis logic is LLM-led. Do not build keyword lists for jargon, fatigue, FOMO, or intent matching. The backend should keep the UI stable when the LLM is unavailable, but semantic judgments belong to the model.

Tasks:
- Keep a valid neutral fallback command for demo resilience.
- Ensure failed, missing, or slow model calls do not crash the UI.
- Keep fallback behavior clearly separate from LLM analysis.
- Do not infer coaching, fatigue, or FOMO with hard-coded words.

Done when:
- The backend always returns a valid `UI Command`.
- Without an LLM, the backend returns neutral.
- With an LLM, coaching, fatigue, and FOMO are generated by the model from the rolling transcript and context.

### 5. Time-Slotted LLM Prompt Chain

Tasks:
- Choose the LLM provider available for the hackathon.
- Build a system prompt that requires concise, actionable UI commands.
- Run model analysis on a fixed cadence, for example every 5 seconds, instead of on every transcript chunk.
- Send only:
  - rolling live transcript window
  - active session metadata
  - audience profile
  - current agenda item
  - next agenda item
- Require structured output matching the `UI Command` schema.
- Add timeout and error handling with neutral fallback.

Prompt constraints:
- Use only transcript text and session metadata.
- Do not invent facts.
- Prefer short action prompts.
- Return neutral when no useful action is needed.

Done when:
- The backend can produce structured commands from an LLM on the configured time slot.
- Bad or slow LLM responses do not break the demo.
- Neutral fallback still works.

### 6. Topic Fatigue HUD Support

Tasks:
- Pass rolling transcript context and agenda fields to the LLM.
- Let the LLM decide if a discussion is lingering too long.
- Include `next_agenda_item` when available.
- Do not detect repeated topics with backend keyword rules.

### 7. FOMO Generator Support

Tasks:
- Pass current transcript context and attendee intent distribution to the LLM.
- Let the LLM decide whether a highly actionable insight exists.
- Generate short summary snippets from model output only.
- Do not detect FOMO snippets with backend keyword rules.

## Backend MVP Priority

1. Session and audience data load successfully.
2. Live transcript ingestion works and maintains history.
3. LLM analysis runs every configured time slot.
4. Neutral fallback keeps the UI valid when the LLM is unavailable.

## Out of Scope for This MVP

- Native real-time audio ingestion.
- Audio-to-audio Realtime API flows.
- Speech rate, intonation, or monotone detection.
- Complex multi-room event operations.
