# Frontend & Backend Integration Plan: Live Event Agentic Assistant

## Goal
Define the data contracts, shared state boundaries, and integration milestones between the Streamlit frontend and the data/LLM backend.

## Shared Interfaces (Data Contracts)

Agree on these interfaces early so both engineers and their AI agents can work independently.

### 1. Session Context
Passed from the backend to the UI when a session is selected.
```python
{
    "session_id": "string",
    "title": "string",
    "room": "string",
    "agenda": ["string"],
    "current_agenda_item": "string",
    "next_agenda_item": "string | None",
}
```

### 2. Audience Profile
Generated by the backend and displayed in the UI's secondary panel.
```python
{
    "session_id": "string",
    "attendee_count": 0,
    "ai_experience_distribution": {
        "Beginner": 0,
        "Intermediate": 0,
        "Advanced": 0,
    },
    "academic_background_distribution": {},
    "top_intents": ["string"],
    "beginner_ratio": 0.0,
}
```

### 3. Transcript Chunk
The standard shape for a piece of transcribed audio flowing into the system.
```python
{
    "timestamp": "00:00:00",
    "speaker": "string",
    "text": "string",
}
```

### 4. UI Command
The ultimate output from the backend analysis layer (either rule-based or LLM-generated) that the UI must render in the Action Zone.
```python
{
    "type": "coaching | fatigue | fomo | neutral",
    "priority": "low | medium | high",
    "headline": "3-7 word action prompt",
    "detail": "short explanation",
    "target": "speaker | moderator | attendee",
    "related_topic": "string | None",
}
```

## State Management

Since we are using Streamlit, backend state and UI state must be managed carefully using `st.session_state`.
1. **`st.session_state.active_session`**: Stores the currently selected `Session Context`.
2. **`st.session_state.transcript_history`**: A rolling list of the last N `Transcript Chunk` objects.
3. **`st.session_state.current_commands`**: A list of the latest `UI Command` objects returned by the backend analysis loop.

## Integration Milestones & Parallel Work

### Phase 1: Static Integration
*   **Backend:** Provides static, hardcoded versions of the shared interfaces (e.g., `get_dummy_audience_profile()`, `get_dummy_ui_commands()`).
*   **Frontend:** Builds the Streamlit layout using these static functions to ensure the UI renders correctly.
*   **Checkpoint:** The UI looks complete but uses fake data.

### Phase 2: Live Data Flow
*   **Backend:** Wires up the actual pandas data loaders and live transcript ingestion. Implements the rule-based analyzer.
*   **Frontend:** Implements the auto-refresh loop (`st_autorefresh` or `st.rerun()`) to pull live transcript chunks and command updates.
*   **Checkpoint:** Selecting a session updates the audience profile dynamically, and live transcripts flow into the UI feed, triggering rule-based Action Zone updates.

### Phase 3: The LLM Brain Hookup
*   **Backend:** Swaps the rule-based analyzer for the LLM prompt chain.
*   **Frontend:** Ensure the UI handles slight delays from LLM API calls gracefully (e.g., showing a subtle loading spinner in the Action Zone if it's "thinking").
*   **Checkpoint:** The Action Zone displays highly intelligent, context-aware prompts.

### Phase 4: Final Demo Integration
*   **Both:** Pick one specific session dataset to use for the hackathon judging.
*   **Both:** Rehearse the "happy path" ensuring the transcript perfectly triggers the Coaching, Fatigue, and FOMO states in the UI. Add manual fallback buttons to the UI just in case the live transcript feed fails during the live demo.
